작업 계획 문서를 읽고 `local/progress-issue-N.md` 체크포인트 파일을 생성한다.
새 세션에서 한 파일만 읽으면 바로 작업을 재개할 수 있도록 모든 컨텍스트를 압축해 담는다.

## Steps

1. **관련 파일 수집** (병렬로 실행)
   - `git branch --show-current` — 브랜치명에서 이슈 번호 추출 (e.g. `fix/issue-4-...` → `4`)
   - `local/` 폴더에서 이슈 관련 파일 탐색:
     - `work-plan-issue-N.md` 또는 사용자가 지정한 작업 계획 파일
     - `review-work-plan-issue-N.md` (있으면)
   - `docs/madr/` 폴더에서 관련 ADR 파일 탐색 (있으면)

2. **파일 읽기** (병렬로 실행)

   위에서 찾은 파일들을 모두 읽는다.

3. **progress 파일 생성**

   `local/progress-issue-N.md` 를 아래 구조로 작성한다:

   ```markdown
   # Issue #N 진행 현황

   > 브랜치: `<branch-name>`
   > 전체 계획: [work-plan-issue-N.md](...) | 리뷰: [...] | ADR: [...]
   > (없는 항목은 생략)

   ---

   ## 핵심 결정 (ADR/작업계획 요약)

   (3~5줄 이내로 핵심만. 새 세션에서 배경을 재파악하지 않아도 될 정도로 압축)

   ---

   ## 리뷰 피드백

   (리뷰 파일이 있으면 필수/권장으로 분류, 이미 반영된 항목은 ✅ 표시)
   (리뷰 파일이 없으면 이 섹션 생략)

   ---

   ## 커밋 진행 현황

   ### ⬜ Commit 1 — `<type>: <description>`

   파일: ...
   구현 스펙:

   - (이 커밋에서 구현해야 할 내용을 구체적으로 — Claude가 임의 해석하지 않을 수준)

   ### ⬜ Commit 2 — ...

   ...

   ---

   ## 핵심 파일 경로

   | 파일 | 역할 |
   | ---- | ---- |
   | ...  | ...  |
   ```

   **작성 원칙:**
   - 새 세션에서 이 파일 하나만 읽으면 바로 다음 커밋 작업을 시작할 수 있어야 한다
   - 원본 파일은 링크만 — 기본은 이 파일로 충분하게
   - 구현 스펙은 구체적으로 (모호하면 Claude가 잘못 구현함)
   - 리뷰 피드백 중 "필수" 항목은 반드시 구현 스펙에 반영

4. **완료 안내**

   생성된 파일 경로를 알리고, 다음 세션 재개 프롬프트 예시를 보여준다:

   > `local/progress-issue-N.md` 확인하고 다음 작업 진행해줘.
